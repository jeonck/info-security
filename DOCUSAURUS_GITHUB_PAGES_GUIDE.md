# Docusaurus + GitHub Pages 구축 가이드

> 실제 구축 경험 기반. 시행착오를 줄이기 위한 실전 가이드.

---

## 빠른 시작 체크리스트

```
□ 1. npx create-docusaurus@latest . classic --typescript
□ 2. 기본 tutorial docs 삭제
□ 3. docs 폴더 구조 생성 (index.md 패턴 사용)
□ 4. autogenerated 사이드바로 먼저 빌드 → doc ID 확인
□ 5. 확인된 ID로 manual 사이드바 작성
□ 6. 홈페이지 링크 수정
□ 7. docusaurus.config.ts GitHub 정보 입력
□ 8. GitHub 저장소 생성 및 push
□ 9. GitHub Pages 활성화
```

## 핵심 시행착오: Docusaurus doc ID 규칙

### 문제

`index.md` 파일을 manual 사이드바에서 참조할 때 ID를 잘못 쓰면 빌드 실패:

```
Error: Invalid sidebar file at "sidebars.ts".
These sidebar document ids do not exist:
- 01-foundations/01-cryptography/index   ← 안됨
- foundations/cryptography               ← 안됨
- 01-foundations                         ← 안됨
```

### 정답: ID 규칙 2가지

| 규칙 | 설명 | 예시 |
|------|------|------|
| 숫자 prefix 제거 | `01-foo` → `foo` | `01-foundations` → `foundations` |
| index.md는 `/index` 포함 | 파일명 그대로 유지 | `index.md` → `.../index` |

**결론적으로 올바른 ID 형식:**

```
docs/01-foundations/01-cryptography/index.md
                    ↓
sidebar ID: foundations/cryptography/index
```

```ts
// sidebars.ts - 올바른 예시
{type: 'doc', id: 'foundations/cryptography/index'}

// 틀린 예시들
{type: 'doc', id: '01-foundations/01-cryptography/index'}  // ❌ 숫자 prefix 제거 안됨
{type: 'doc', id: 'foundations/cryptography'}              // ❌ /index 빠짐
{type: 'doc', id: '01-foundations/01-cryptography'}        // ❌ 둘 다 틀림
```

### doc ID 확인하는 확실한 방법

1. `sidebars.ts`를 임시로 autogenerated로 변경:
```ts
const sidebars: SidebarsConfig = {
  mySidebar: [{type: 'autogenerated', dirName: '.'}],
};
```

2. `npm run build` 실행 (성공해야 함)

3. `.docusaurus/globalData.json` 파싱:
```bash
cat .docusaurus/globalData.json | python3 -c "
import json, sys
d = json.load(sys.stdin)
versions = d['docusaurus-plugin-content-docs']['default']['versions']
for doc in sorted(versions[0]['docs'], key=lambda x: x['id']):
    print(doc['id'], '->', doc['path'])
"
```

4. 출력된 ID를 manual 사이드바에 그대로 사용

## frontmatter `id:` 필드 주의사항

### 문제

`index.md`에 `id: something`을 설정하면 Docusaurus가 해당 doc의 ID를 변경.  
sidebar에서 다른 ID로 참조하면 "does not exist" 오류 발생.

### 해결

**manual 사이드바를 쓸 때는 frontmatter에 `id:` 를 쓰지 않는다.**

```md
---
title: 암호학          ← OK
sidebar_label: 암호학  ← OK
sidebar_position: 1    ← OK
id: cryptography       ← ❌ 제거할 것 (ID가 바뀌어 sidebar 참조 깨짐)
---
```

이미 작성했다면 일괄 제거:
```bash
find docs -name "index.md" -exec sed -i '' '/^id: /d' {} \;
```

## 기본 tutorial 삭제 후 홈페이지 링크 수정

기본 생성된 사이트는 `/docs/intro`로 링크되어 있음.  
tutorial 삭제 후 이를 수정하지 않으면 빌드는 성공해도 링크가 404.

```tsx
// src/pages/index.tsx - 수정 필요
// Before
to="/docs/intro"

// After (실제 첫 번째 doc 경로로)
to="/docs/foundations"   // 또는 본인 첫 페이지 URL
```

```ts
  // ...

```

### onBrokenMarkdownLinks 경고 처리 (Docusaurus 3.10+)

최상위에 쓰면 deprecated 경고 발생:
```ts
// ❌ deprecated
onBrokenMarkdownLinks: 'warn',

// ✅ 올바른 위치
markdown: {
  hooks: {
    onBrokenMarkdownLinks: 'warn',
  },
},
```

## 권장 docs 폴더 구조

숫자 prefix로 순서를 제어하되, 각 카테고리에 `index.md`를 두어 개요 페이지로 활용:

```
docs/
├── 01-category-a/
│   ├── index.md              ← 카테고리 개요 (sidebar link 대상)
│   ├── 01-subtopic/
│   │   └── index.md
│   └── 02-subtopic/
│       └── index.md
└── 02-category-b/
    ├── index.md
    └── ...
```

**URL 결과:** `/docs/category-a`, `/docs/category-a/subtopic` (숫자 prefix 자동 제거)

---

## sidebars.ts 패턴 (index.md 구조 기준)

```ts
import type {SidebarsConfig} from '@docusaurus/plugin-content-docs';

const sidebars: SidebarsConfig = {
  mySidebar: [
    {
      type: 'category',
      label: '카테고리 A',
      link: {type: 'doc', id: 'category-a/index'},       // ← /index 포함
      items: [
        {type: 'doc', id: 'category-a/subtopic-1/index', label: '소주제 1'},
        {type: 'doc', id: 'category-a/subtopic-2/index', label: '소주제 2'},
      ],
    },
  ],
};

export default sidebars;
```

## GitHub Actions 배포 워크플로우

`.github/workflows/deploy.yml`:

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches:
      - main

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: build

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/deploy-pages@v4
        id: deployment
```

## GitHub 배포 명령어 순서

```bash
# 1. 저장소 생성
gh repo create USERNAME/REPO_NAME --public --description "설명"

# 2. git 초기화 및 push
git init
git add .
git commit -m "initial commit"
git remote add origin https://github.com/USERNAME/REPO_NAME.git
git push -u origin main

# 3. GitHub Pages 활성화 (GitHub Actions 방식)
gh api repos/USERNAME/REPO_NAME/pages --method POST --field build_type=workflow

# 4. 배포 상태 확인
gh run list --repo USERNAME/REPO_NAME --limit 3
gh run watch RUN_ID --repo USERNAME/REPO_NAME
```

> **주의:** GitHub Pages 설정 > Source를 "GitHub Actions"로 바꾸는 게 UI에서는 직관적이지만, `gh api`로 `build_type=workflow`를 설정하면 동일 효과.

## 전체 시간 절약 요약

| 단계 | 시행착오 원인 | 해결책 |
|------|-------------|--------|
| doc ID 파악 | 규칙 불명확 | autogenerated 빌드 후 globalData.json 확인 |
| frontmatter id | 충돌 발생 | manual sidebar 쓸 때 frontmatter `id:` 제거 |
| 홈페이지 링크 | 기본값이 삭제된 페이지 | 첫 번째 실제 doc 경로로 수정 |
| onBrokenMarkdownLinks | deprecated 경고 | `markdown.hooks` 하위로 이동 |
| GitHub Pages 활성화 | UI vs CLI | `gh api ... --field build_type=workflow` |
